# 产品想法记录

本文档记录当前讨论中的产品方向与交互想法，用于后续设计和实现时参考。以下内容不代表已经实现，也不代表最终方案已经冻结。

## Agent 协作方式

在需要使用 Cloud Code 协作开发时，默认先下载并安装供 Cloud Code 使用的 Codex 官方 Plugin。规划、方案设计、代码审查、风险分析和验收检查优先通过该 Plugin 完成，实际代码编写与修改交由 Cloud Code 执行。

默认要求是：规划阶段不直接改代码，代码审查阶段不直接改代码，只输出实现计划、问题列表、修改建议和验收结论。

## 启动体验

启动图不做成复杂首页，也不依赖真实菜品流。Launch Screen 保持静态、简洁、绿色主题，强调品牌识别和食物氛围。

启动后的首屏过渡页可以承担视觉吸引任务。这里可以使用精选菜品图、本地插画或统一风格的食物主视觉，在进入主界面前快速建立产品气质。

过渡页的文字应尽量少。优先保留品牌名、厨房名或一个简短动作入口，不默认堆叠副标题和说明文案。

## 图片上传与处理

菜品图片的核心目标是快速记录，而不是要求用户先拍出完美照片。用户在做好菜后，可以直接拍照上传，或者从相册中选择图片。

第一原则是减少操作步骤。上传链路应尽量简化为选择图片后直接上传原图，不引入强依赖的手动裁剪流程。

图片优化主要交给服务端处理。后端可结合大模型或图像处理流程，对原图进行主体识别、背景提纯、轻量增强和统一风格处理。

系统应保留原图兜底。即使处理失败，也不影响菜品正常显示。

长期看，图片资源格式可逐步向 AVIF 收敛，以兼顾体积与画质。第一阶段不强制前端只上传 AVIF，可由服务端统一转码。

## 语音输入

语音应成为高频入口，而不是附属功能。用户可以通过语音快速描述菜品信息，后端把自然语言解析成结构化字段。

菜品录入场景下，语音输入用于生成菜名、分类、食材等字段。用户说一句话即可完成基础录入，再做少量确认或修正。

首页也可以提供语音下单入口。用户直接说想吃什么菜，系统解析后生成购物车或订单内容。

这两个语音入口应分开设计。一个用于录入菜品，一个用于发起点单，它们的解析目标和交互反馈不同。

## 权限与系统能力

后续需要补齐与功能匹配的系统权限说明，包括相机、相册、麦克风和语音识别。权限说明文案应短，只解释直接用途，不写泛泛描述。

iOS 不存在单独的联网权限弹窗。涉及联网、模型处理、跨设备同步等能力时，主要在产品说明和状态反馈中体现，而不是等待系统层授权。

跨设备同步属于后续重要方向。图片、菜品、订单和语音解析结果都应以多设备一致为前提设计数据结构与接口。

隐私与上架相关事项不应只停留在人工记忆层。凡是会重复出现、容易遗漏、会影响上架、合规或提审质量的工作，都应尽早沉淀为脚本、清单或 CI 检查。

这类事项包括但不限于权限说明文案、隐私清单、App Store Connect 元数据、截图尺寸和命名、构建与打包检查、版本号管理、发布前校验和文档同步。原则上，同一提醒出现第二次后，就应进入自动化范围。

## 贴纸与视觉元素

未来可以加入贴纸或轻量装饰元素，增强菜品图片和页面的识别度。但贴纸不应干扰主路径，优先级低于快速上传、快速录入和快速下单。

贴纸更适合作为图片处理完成后的附加表达，或作为某些精选菜品的视觉层，而不是所有页面默认铺开。

## 界面表达原则

界面内文字保持克制。默认只保留必要标题、操作按钮和状态信息，不主动增加副标题、解释文字和冗余描述。

产品应更像一个直接可用的厨房工具，而不是带大量说明文案的展示型页面。复杂理解交给系统处理，用户界面只暴露最少必要信息。

界面设计默认降低认知负担。文案要直接，相关信息要成组出现，次要细节不要抢占主路径，当前最重要的内容应通过位置、层级和视觉重点被快速识别。

系统应识别用户是否犹豫、陌生或处在高风险操作中，并提供脚手架式帮助，例如上下文提示、合理默认值、字段示例、可恢复操作和必要时的阶段性引导。帮助应是临时支撑，而不是长期打断，熟悉后应自然退场。

流程设计应优先保持连续流体验。任务目标要清楚，步骤要短，反馈要及时，难度要与用户当前能力匹配，避免无聊的重复劳动，也避免因为信息过多、跳转过深或反馈滞后而产生挫败。

产品沟通应明确传达长期维护者视角。收费、无广告、隐私克制和评价成本都应在合适场景下被清楚解释，避免让用户误以为产品依赖广告变现或以牺牲数据为代价换取低价。

当某个操作、校验或协作动作第二次以“不要忘记”的形式出现时，不应继续依赖记忆。个人提醒应尽快升级为共享脚本、规范或 CI 约束，让流程靠系统而不是靠自觉维持。

## 测试与可访问性

测试不是上线前附加的步骤，而是界面和数据结构设计时就要考虑的约束。Model、Store 和业务规则优先通过单元测试覆盖，关键用户路径通过 UI 测试逐步补齐。

UI 测试基于系统可访问性树定位和驱动界面元素，因此关键控件必须从一开始就具备稳定、可预期的可访问性语义。用户可见文本服务理解，测试定位则依赖稳定的 `accessibilityIdentifier`，二者不要混用。

主按钮、表单字段、sheet、toast、错误 banner、列表项、删除与撤销动作，都应优先考虑可访问性命名和测试定位能力。界面应默认做到可测、可访问、可恢复，而不是等功能完成后再补。

## 组件与动效

界面应优先建立在 SwiftUI 原生能力之上，再通过自定义组件进行样式和交互封装，而不是绕开原生体系堆叠零散特效。自定义组件的目标是统一视觉、统一交互、统一可访问性和统一测试接口。

按钮、卡片、输入区、数量编辑器、状态标签、反馈提示和 sheet 容器，适合作为第一批长期复用的基础组件。页面应尽量组合组件，而不是在每个页面重新拼接局部样式与局部状态。

动画应作为组件默认行为的一部分被提前设计，包括按压反馈、状态切换、插入删除、sheet 展开收起和 toast 出现消失。默认优先使用 SwiftUI 原生动画和系统过渡，避免脱离平台语义的复杂自定义动效。

后续可以单独建立一组加载与状态反馈组件，用来统一 loading 页、状态徽章、状态卡片和订单项状态切换。这里的重点不是做炫技特效，而是把“正在处理”“正在制作”“已完成”这些厨房语义，转换成连续、可信、可恢复的视觉状态。

### Loading 页与局部 Loading

loading 不应只有一种形式。全屏 loading 页用于启动后首屏过渡、语音解析等待、图片上传处理和大任务提交后的短暂停留；局部 loading 用于菜品卡片、订单项、按钮和 sheet 内的局部处理中。两者应共用一套视觉语言，但节奏不同。

loading 页可以使用居中的主徽章或主卡面作为视觉主体，而不是使用单独的阻塞式 spinner。主徽章建议是一个具有轻体积感的状态球、锅盖徽章、蒸汽徽章或菜品处理徽章，配合柔和脉冲、低频 shimmer、缓慢旋转高光和轻微景深阴影，形成“界面还活着，正在推进”的感觉。

局部 loading 更适合使用骨架屏、shimmer、按钮内部进度环、状态点脉冲和局部高亮扫光，而不是把整个页面盖暗。用户应继续感知当前上下文，而不是被突然打断。

视觉上，loading 动画应强调呼吸感和连续流动感。推荐的基础词典包括柔和脉冲、细窄 shimmer、高光扫过、缓慢热旋转、轻微浮动、进度环填充、辉光扩散和完成瞬间的小幅回弹。这些基础动作可组合使用，但每个场景最多保留一到两个主动作，避免同时闪烁、旋转、缩放导致廉价感。

### 状态切换动效

订单项从准备中到制作中到制作完成，不应只是图标瞬间替换。更理想的方式是把状态切换做成一个独立的过渡容器，让用户感知到“同一个对象正在变成下一种状态”，而不是“旧图标消失，新图标出现”。

推荐把状态容器抽象成状态徽章或状态卡面。外层负责 3D 翻面、卷动、透视、缩放和阴影重心变化，内层承载具体状态动画，例如 preparing 的脉冲和 shimmer，cooking 的热旋转、热气、进度环或锅内搅动，done 的轻微弹起、辉光扩散和稳定落位。这样状态本体动画与状态切换动画可以分离，复用性更强。

“准备中 -> 制作中”适合使用带一点卷动感的翻面或滚动切换，像卡面向前轻翻后露出新的 cooking 徽章。“制作中 -> 完成”适合使用更轻的竖向抬起、定格和高光扩散，表达任务已经落定，而不是继续翻转。危险或失败态不应沿用完成态的上扬节奏，应该改用轻微下沉、去饱和或短促抖动。

### 卡面、徽章与翻面语言

状态表达建议优先使用卡面和徽章两种载体。卡面适合承载订单项、菜品处理进度和多字段上下文；徽章适合承载单一状态、列表内状态点和按钮尾部状态提示。

卡面翻面适合在信息结构变化明显时使用，例如订单项从“准备食材”切到“正在烹饪”时，背面可以带出新的主图标、主文案和辅助进度。徽章翻面适合高频状态切换，因为面积更小、成本更低，也更容易做出真实的 3D 感。

所谓“真实的类 3D 动作”，重点不是做复杂建模，而是同时处理好透视、缩放、遮挡、阴影位置、边缘高光和速度曲线。翻面开始时应先轻微收缩并偏移重心，中段转体最快，结束时带一点阻尼和落位，这样才会像一个有厚度的实体。

### 动效词典建议

preparing 状态适合使用柔和脉冲、细 shimmer、轻微呼吸辉光和低速扫光，表达“系统已接单，正在准备”。

cooking 状态适合使用持续但不过快的旋转、热气上浮、局部粒子漂移、进度环推进和中心主体的小幅弹性，表达“任务正在进行且有热量感”。

done 状态适合使用轻微弹起、辉光扩散、勾形或完成图标稳定落位，以及从活跃动画切换为安静静态的收束过程，表达“已经完成，不再持续占用注意力”。

loading 页适合使用主徽章慢呼吸、背景高光缓慢流动、卡片层次轻微浮动和内容骨架渐进显现，表达“页面正在搭建而不是卡住”。

### Lottie、SVG 与实现取舍

未来如果评审后允许引入第三方，可以把 Lottie 作为复杂状态动画资源的承载方式之一，尤其适合设计先行、需要多人协作产出统一动画资产的场景。它的优势是动画细节丰富，状态本体资源可由设计师迭代，客户端主要负责播放、暂停和切段切换。

即便允许使用 Lottie，也不建议把所有动效都做成独立的 Lottie 文件然后直接硬切。两个动画资源之间如果没有统一的过渡容器，切换时仍然会显得像两个世界拼起来。更合理的做法是让 Lottie 负责状态本体，让外层 SwiftUI 或 Core Animation 负责翻面、卷动、缩放、透视和切换时序。

不建议把 SVG 动画作为 iOS 主方案，主要原因不是视觉不够好，而是接入链路和运行时一致性不理想。SVG 动画在 iOS 原生体系里缺少稳定统一的高质量运行时，很多效果最终要依赖 WebView、自定义解析或转码，状态同步、性能调优、命中测试、资源管理和离线一致性都会变复杂。

SVG 更适合作为静态矢量源文件或设计交换格式，而不是项目中的主运行时动画格式。如果设计阶段产出的是 SVG 动画，落地时更适合转成 Lottie、序列帧、视频纹理，或者直接改写成 SwiftUI / Core Animation 可控动画。

### 结构化实现建议

后续实现时，可以把这套能力拆成三层。最外层是状态切换容器，负责翻面、卷动、透视、缩放和切换节奏；中间层是卡面或徽章样式层，负责形体、材质感、阴影和高光；最内层是具体状态内容层，负责 preparing、cooking、done 各自的动画资源与文案。

这种结构的好处是 loading 页、订单项状态徽章、菜品卡片和按钮尾部状态提示都可以复用同一套切换语言，只替换内容层即可。页面看到的是统一的产品气质，实现层得到的是可测试、可替换、可逐步升级的组件结构。

## 上架素材与市场自动化

上架工作不应等到产品完成后才临时处理。截图、预览视频、隐私政策、App Store Connect 元数据、更新日志和宣传文案，都应提前进入可重复执行的素材生产流程。

截图与预览视频应先服务于“说明产品价值”，而不是追求广告片质感。第一阶段先固定首发信息架构，再围绕最关键的 5 到 7 个场景生成素材，例如首页、核心录入流程、语音入口、菜品详情、订单状态变化和图片处理结果。

截图生产应尽量半自动化。设备型号、系统版本、主题、语言、测试数据、账号状态和页面入口都应固定，避免每次发布重新手工整理。AI 适合负责截图标题、副文案和版本变体，人负责最终挑选和审美裁决。

预览视频也应走低成本、可重复的生产路径。第一阶段只做 15 到 30 秒的短预览，重点展示进入页面、执行关键操作、看到状态变化和完成结果，不默认做重包装宣传片。录制链路可基于 Simulator 或真机录屏，后期只保留必要裁切、字幕和片头片尾。

更长期的目标不是一次性做完素材，而是建立素材流水线。建议把这一层拆成固定设备与测试数据、截图脚本、录屏脚本、字幕与文案模板、导出目录规范和发布检查清单。这样每次版本发布时，素材更新就不再依赖临时记忆。

市场宣传也应按“内容系统”而不是“单次写稿”来设计。最适合自动化的是 App Store 文案、版本更新说明、多平台短帖、小红书或微博标题变体、FAQ、视频字幕稿和产品介绍长文的初稿生成。

这类自动化内容应建立统一输入结构，例如本次版本新增能力、目标用户、核心使用场景、与上一版相比的变化、希望强调的产品气质。AI 根据这些输入输出多套文案草稿，再由人进行筛选、删改和定调。

不适合完全自动化的部分，主要是品牌判断、渠道选择、节奏控制、视觉品味和最终发布决策。自动化应优先覆盖重复劳动和格式化产物，不应替代产品定位与审美判断。

因此，上架准备的优先级不只是补文案，而是先建立稳定的素材生产流水线。只有截图、视频、元数据和宣传内容的源头可以稳定生成，后续市场动作才有连续性，自动化也才真正有价值。

## 海外宣传平台与自动化边界

海外宣传平台不应只按国家划分，更适合按内容类型划分。短视频触达优先投放 Instagram、TikTok 和 YouTube Shorts；产品正式发布适合 Product Hunt；持续更新、独立开发过程和版本进展适合 X；真实讨论、反馈收集和问题切入更适合 Reddit。

这类平台矩阵的目标不是一次性铺满所有平台，而是让同一份核心内容可以被拆成不同格式重复利用。例如同一个版本更新，可以拆成短视频脚本、字幕稿、Product Hunt 标题与简介、Reddit 帖文草稿、X 线程和评论回复草稿。

自动化在这里的含义，应明确限定为重复劳动自动化，而不是无人值守的全自动营销。AI 可以负责生成文案初稿、短视频分镜、字幕、封面标题、平台变体文案、FAQ 和更新摘要，也可以负责从候选话题中筛选更适合参与的讨论串。

短视频本身也可以部分自动化。系统可以根据版本变更、核心卖点和目标用户生成脚本、镜头顺序、口播稿和字幕，但成片生成仍应视工具链决定。第一阶段应把重点放在脚本、录屏素材、字幕和导出规范，而不是追求完全自动生成视频。

对外发布动作应保留人工审核层。尤其是 Reddit、X 回复、社区评论参与和热点话题跟进，不应直接做成全自动发布。更合理的方式是系统先生成候选回复、附带上下文摘要和风险提示，再由人确认是否发送。

官方 API 能力可以作为执行层接入，但不应改变前面的审核原则。X 官方 API 支持发帖和回复，Reddit 官方接口也支持帖子与评论读写，因此技术上可以接入“草稿生成 -> 人工确认 -> API 发布”的链路。Product Hunt 更接近低频正式发布场景，适合自动准备发布素材和联动文案，而不是高频自动分发。

这意味着市场自动化系统更适合拆成五层。第一层是内容源头，例如版本更新、功能亮点、用户反馈和截图素材。第二层是 AI 生成层，负责产出多平台草稿。第三层是候选发现与排序层，负责收集适合参与的话题和社区入口。第四层是人工审核与定调层。第五层才是通过官方 API 或人工界面完成最终发布。

整体原则应保持克制。自动化的目标是放大内容生产效率、减少重复格式化劳动、降低跨平台分发成本，而不是用机器人式行为透支账号信誉和社区关系。

## 上架合规素材的合适做法

上架相关素材应优先采用“成熟工具生成基础版本，产品侧统一校准”的方式，而不是全部从零手写。图标、logo、隐私政策初稿、权限说明、App Store 元数据和公开隐私网页，都适合先用现成工具或模板快速生成，再根据真实产品行为做最后收敛。

图标和 logo 适合先通过成熟设计工具或 AI 草图工具探索方向，但最终版本必须经过产品侧人工定稿。判断标准应优先看小尺寸识别度、品牌一致性、与竞品的区分度和不同背景下的稳定性，而不是只看首屏视觉效果。

隐私政策不应只作为提审时补交的文件，而应作为产品数据流的外部说明。更合适的方式是先根据登录、账号、图片上传、语音输入、支付、埋点、第三方 SDK、跨设备同步和客服反馈这些能力，整理出真实数据流，再生成隐私政策、App Privacy 标签填写稿和公开网页内容。

权限说明文案不应泛泛而谈。每一项系统权限都只解释直接用途，例如为什么需要相机、麦克风、相册、通知或语音识别，而不应混入营销语言。权限说明、隐私政策、App Privacy 标签和产品实际行为必须保持一致。

公开隐私链接应尽早稳定下来，不要等到提交 App Store Connect 时再临时挂一个页面。更合适的方式是维护一份可长期访问的隐私网页，并让 App Store、官网和产品内入口共用同一来源。

整体上，上架合规素材最适合收敛为一套统一资产包。该资产包既服务提审，也服务版本更新、网站同步、法务校对和团队协作，避免每次发布重复收集和重新整理。

## 本地缓存与数据新鲜度（未来想法）

当前每次启动应用，菜品图片和数据都会重新从网络拉取，首屏存在明显加载等待。期望的体验是：如果数据没有变化，首屏直接展示上次的本地缓存，无需等待；如果有变化，后台静默拉取并与本地对比后再更新 UI。

推荐策略是 stale-while-revalidate：首屏立即渲染本地缓存数据与图片，同时后台发起带条件的刷新请求，命中则无感替换，未命中则完全不扰动界面。

服务端与客户端需要配合的数据新鲜度判定方式，按成本从低到高：

1. ETag / Last-Modified：服务端返回版本头，客户端下次请求携带 `If-None-Match`，未变则 304 无 body，图片与 JSON 均适用。
2. `updated_at` 增量：列表类接口支持 `?since=<timestamp>`，只返回变更过的行，适合菜品和订单列表。
3. 内容哈希 URL：图片 URL 内嵌内容哈希（例如 `dish-abc123.jpg`），内容变更即 URL 变更，缓存天然失效，旧 URL 可安全保留。
4. 推送 / 实时通道：WebSocket、SSE 或静默推送主动通知客户端变更，新鲜度最强，基础设施成本也最高。

客户端侧需要建立两层缓存：图片走按 URL / 哈希的磁盘缓存，数据走本地持久化（菜品、订单最近一次快照）。首屏读缓存即渲染，后台请求带上 ETag 或 `since`，拿到新数据后再 diff 更新 Store，SwiftUI 自动重渲染，用户最多感知到极短暂的旧数据，而不会再看到空白骨架。

这一方向和前文「Loading 页与局部 Loading」一致：全屏阻塞式加载应尽量避免，界面默认就应是活着并可交互的。
